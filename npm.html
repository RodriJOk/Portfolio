Introduccion a Npm - Guia basica

En este ultimo tiempo, hemos visto como Node a tomado mucha popularidad en el mundo del desarrollo en el lenguaje Javascript. Lo interesante es que no podemos hablar de Node.Js y pasar por alto NPM. Es por eso que en este articulo hablaremos de el a profundidad. Espero que te sea de mucha utilidad. üòä

NPM - ¬øQue es?

NPM es un gestor de paquetes üì¶, el mas popular que tiene Javascript, donde encontaras una gran cantidad de recursos para poder implentar en tus proyectos. Tambien vas a poder crear tus propios paquetes y compartilos con toda la comunidad. En su sitio podras ver a detalle mas informacion sobre los paquetes que podras implementar en tus proyectos. Desde librerias open source hasta grandes proyectos. NPM (por sus siglas en ingles Node Package Mananger) se encarga de administar todos los paquetes y, algo importante que debemos tener en cuenta, es que para trabajar con NPM, debemos tener instalada la version de Node.

Pero, ¬øComo hacemos eso?. Tranquilo/a üòâ. Es muy simple, aqui veras una guia de pasos para llevar a cabo.

En Windows, sigue estos pasos:

1_ Descarga e instalacion de Node: Para poder empezar a trabajar con NPM, necesuitamos tener instalado Node.js en nuestro computador. Para ello vamos a su sitio oficial y descargamos Node.

Una vez descargado Node, pasaremos a la etapa de instalacion. (El famoso Next -> Next üòÖ). Una vez terminada la instalacion, podemos chequear si tenemos Node en nuestras computadoras si habriamos la consola (Simbolo del sistema) y podemos ejecutar el comando node -v. Si como resultado, vemos el numero de la version de Node.js, lo habras instalado correctamente üôåüèª.

Tambien podemos comprobar que tenemos instalado NPM, si ejecutamos el comando npm -v y podemos ver la version del mismo

Si queremos asegurarnos de que tenemos la ultima versiom podemos ejecutar el comando npm install -g npm@lastest

Inicia un proyecto con npm

Es de mucha importancia de que cuando vayas a crear un proyecto, lo inicialices con npm. Esto nos va a permitir generar una cierta configuracion con ciertos datos importantes para nuestro entorno, asi como tambien compartirlo con otros desarrolladores y puedan tener una nocion de lo que esta ocurriendo, para eso nos vamos a mover a nuestra terminal, en donde vamos a inicilaizar nuestro primer proyecto con NPM. Como muestra de ejemplo, voy a crear una carpeta con el nombre NPMjs y voy a posicionarme dentro de el con el comando cd "nombrearchivo".

Una vez creada nuesta carpeta del proyecto, ahora si estamos listos para poder inicializar nuestro proyecto con NPM. Pero ¬øcomo? ¬°Es muy sencillo! Solo debemos ejecutar el comando npm init. Al ejecutar este comando, nos va a permitira nosotros, crear el archivo package.json, con el cual vamos a tener una configuracion establecida, una descripcion del proyecto y cierto valores que necesita.

Al ejecutar el comando tambien nos aparecera una peque√±a introduccion, con algunas explicaciones. Y tambein nos permitira ir completando la informacion con los valores que desemos (con algunas sugerencias).

Los valores que vamos a tener que configurar son los siguientes

  1. Nombre del paquete (package.name): Aqui debemos colocar el nombre de nuestro proyecto

  2. Version: Numero de version de nuestro proyecto. La sugerencia es la version inicial (1.0.0)

  3. Descripcion (Description): En este campo podemos colocar una breve descripcion de nuestro proyecto

  4. Punto de entrada (entry point): Este paso es importante, por que aqui vamos a definir nuestro punto de entrada de nuestro proyecto.

  5. Test command: Aqui nosotros podremos configurar comandos de utilidad para nuestro proyecto.

  6. Repositorio de Git (git): Aqui podemos establecer el nombre de nuestro repositorio de Git

  7. Palabras claves (keywords): Aqui podremos establecer palabras claves que tengan relacion o describan a nuestro proyecto.

  8. Autor (Author): Aqui podremos establecer quien es la persona que esta construyecndo este proyecto inicialmente.

  9. Licencia (license): Aqui podremos configurar la licencia que tendra nuestro proyecto. En este aspecto podremos elegir entre MIT o ISC.

Luego de establecer los valores, tendremos un resumen con todas las configuraciones que hemos establecido. Si luego de darle un vistazo esta todo correcto, podemos terminar la configuracion de nuestro proyecto.

Luego de terminado todos estos pasos, vamos a poder ver en nuestro proyecto un archivo llamado package.json

Instalacion de dependencias

Veamos como podemos instalar paquetes, desde un paquete convencional hasta uno de forma global, o con todas las opciones que tenemos para instalar paquetes dentro de tus proyectos. Quizas no utilices todas cuando este contruyendo un proyecto, pero que entiendas que existen, te van a yudar para obtener contexto de como trabajar con estos proyectos. Siguiendo con nuestro proyecto NPMjs, dentro de el vamos a crear una carpeta, llamada scr. Recordemos que los comandos que nos ofrece la consola para crear una carpeta es mkdir nombrecapeta (mkdir, del ingles make directory). En este caso, el comando que utilizaremos es mkdir scr. De esta forma creamos la carpeta, pero todavia no estamos en ella, por eso, para esto podemos ejecutar el comando cd nombrecarpetaamover (cd, del ingles change directory). En nuestro caso, ejecutamos el comando cd scr. Una vez realizado esto, dentro de nuestra carpeta src, podemos crear nuestra carpeta index.js. Para poder crear archivos, podemos utilizar el comando touch index.js. Partiendo de esto y de que ya estamos en la raiz del proyecto, aqui es donde nosotros vamos a empezar a instalar las dependencias. Estas dependencias son recursos que vamos a utilizar dentro de nuestro proyectos, partiendo de que hay una gran cantidad de estos recursos.

Imaginemos ahora que queremos instalar un paquete en nuestro proyecto que se encarga de manejar las fechas. Este paquete se llama moment. ¬øPero como lo podemos instalar? Es muy simple, lo podemos instalar con el comando npm init nombreDelPaquete. Algo interesante que debemos tener en cuenta es que a partir de la version 5 de npm, debiamos instalar la dependencia con el flag --save. Si continuamos con el ejemplo anterior, antes debiamos instalar moment asi : npm install moment --save. Pero luego de la version 5 de NPM, el flag --save se ejecuta por defecto, sea que lo pongamos explicitamente o no. ¬øPero que significa --save? Significa que este paquete que vas a instalar dentro del proyecto es necesario para que se encuentre en produccion. Entonces debemos de tener cuidado un paquete que estemos instalando es a produccion y cuando no lo es. De esta forma podemos identificar otro flag, que es --save-dev. Este flag nos va a permitir a nosotros establecer que este paquete que vamos a instalar solo es necesario en nuestro entorno local o en el entorno de desarrollo. Partiendo de esta diferencia, nosotros vamos a tomar decisiones cuando vamos a estar instalando las dependencias que requiere nuestro proyecto. Por que es muy importante no mandar dependencias a produccion, ni omitir algunas que deban estar en produccion. Partiendo de esto vamos a proceder con la instalacion.

De esta manera podemos instalar dependencias en nuestro proyecto. Cuando nosotros instalamos una dependencia, nuestro equipo hace una conexion con los servidores de NPM, extrae la informacion que necesita, crea unos archivos necesarios y despues nos da un output con lo que ha sucedido. Ha instalado moment en la version 2.29.1. Y tambien vamos a ver cambios en nuestro archivo. Vamos a poder ver nuestra famosa carpeta de node_modules, y dentro podremos ver la carpeta de moment.

Recordemos que en la carpeta node_modules, se van a instalar los modulos que tu estas agregando a tu proyecto. Esta carpeta es importante para que tu proyecto funcione, pero no debe ser enviada a ningun repositorio, sino que debemos ignorarla. Esto lo podemos hacer creando un archivo con la extencion .gitignore. Dentro de esta carpeta, debemos colocar node_modules/

De esta forma, definimos que la carpeta no debe ser enviada a ningun repositorio, ni a tu proyecto en produccion

Por otro lado, podemos ir al archivo package.json, y vamos a poder notar que tenemos instalada la dependencia moment.

Tambien podemos instalar dependencias de desarrollo. Recordemos que aqui debemos usar el flag --save-dev. Esto lo podemos hacer de la siguiente manera:

De esta manera podremos instalar la dependencia Date-fnd en nuestro proyecto. Algo que es interesante notar es lo que ocurrio en nuestro archivo package.log

En nuestro package.json, lo que notamos es que se agrego una nueva seccion llamada "devDependencies" con nuestro paquete "date-fns". Y esta es una forma de separar nuestras dependencias, con las dependencias de desarrollo

Instalacion de dependencias a nivel global

En algun momento te vas a topar con un paquete que debe ejecutarse y funcionar de forma "global". ¬øPero que significa esto? Cuando nosotros trabajamos con un proyecto, podemos instalar el paquete de moment, y moment puede ser utilizado dentro del contexto de mi proyecto. Pero no esta disponible en otros proyectos. Uno de los principales paquetes que utilizamos de forma global se llama nodemon. Nodemon nos permite generar que este a la "escucha" algun cambio o valor, y va a mentener nuestro proceso a algun comando que estemos ejecutando con Node. Entonces este paquete es requerido de forma global y de esta forma, nosotros debemos de instalarlo con un flag, y en este caso vamos a hacer el ejercicio, y en nuesta terminal vamos a instalar un paquete global. Un detalle importante a tener en cuenta es que cuando instalamos de forma global un paquete, lo debemos hacer con los permisos de administrador correspondientes. Esto lo instalamos con el comando npm install -g nodemon.

Si queremos ver si tenemos el paquete dentro de la lista de recursos que estan instalados de forma global en npm, podemos ejecutar el comando npm list -g --depth 0.

Instalacion de dependencias con force

Continuando con la instalacion de dependencias en nuestro proyecto hay un flag que vamos a utilizar para distingir que esta dependencia no va a ser instalada dentro del proyecto, pero si queremos ver el output que nos retorna. El comando npm install nombreDelPaquete --dry-run simula que instala una dependencia, nos da el output de lo que esta sucediendo, que version instalaria y la cantidad de tiempo que demoraria en intalar los paquetes. Esto es muy util cuando estamos dudando entre si instalar o no.

Tambien contamos con el flag --force el cual nos va a permitir a nosotros, instalar este paquete forzando esa instalacion a que sea desde el ultimo recurso desde el servidor de npm. Este flag tambien tiene un atajo: npm install nombreDelPaquete -f.

Imaginemos que ahora, queremos instalar una dependencia, pero en este caso, conozco y se que necesito una version especifica. En este caso, nos sera de ayuda ejecutar el comando npm install nombreDelPaquete@version.

Actualizar y eliminar paquetes.

Ahora vamos a revisar como podemos actualizar o darnos cuenta de que hay actualizaciones de estos paquetes. El comando que nos permite hacer esto es npm outdate. Al ejecutar este comando aparecera en forma de lista los paquetes que faltan actualizar.

Tambien hay otro comando que nos va a permitir listar los paquetes que tiene el proyecto npm list.

Si queremos actualizar todos los paquetes en los que estemos trabajando en nuestro proyecto, podemos ejecutar el comando npm update.

Si queremos actualizar solo un paquete, a la ultima version, podemos utilizar el comando npm install nombreDelPaquete@latest.

Podemos eliminar un paquete de node_modules y del archivo package.json con el comando npm uninstall nombreDelPaquete.

Tambien disponemos del comando npm uninstall nombreDelPaquete --no-save. Este comando nos permite desistalar un paquete de node_modules, pero no de nuestro archivo de package

Uso de simbolos ^ (carrets) y ~ (virgurilla)

Antes de hablar sobre este punto en concreto, podemos hablar sobre la forma de manejar versionados

Nosotros, normalemnte tenemos tres valores. El primero major: Siempre va a ser un cambio muy importante. En segundo lugar, tambien vamos a tener cambios menores o minor: Estos son cambios que a√±aden ciertas funcionalidades, pero que no son un gran paso para decir que es una version nueva. Y en tercer lugar, nos encontramos con los patch: Aqui encontramos cambios muy menores como bug fixes o algunos parches. Miremos un package-lock de un proyecto

Como bien podemos ver en esta ultima imagen, puede que nos encontremos con un archivo package-lock, que contenga algunos simbolos (^) o (~). Es por eso que es importante saber lo que representan ellos. En primer lugar, si mantenemos el ^ (carrets) dentro de la configuraci√≥n de nuestro package estamos garantizando que cuando realicemos una actualizaci√≥n o tengamos un cambio que podamos realizar, vamos a hacer actualizaci√≥n de los cambios menores y de los parches o bug fixes. Para quedarnos en una sola versi√≥n, podemos eliminar el caret. En segundo lugar, si mantenemos la ~ (virgurilla) establece que vamos a recibir actualizaciones o cambios solamente de los cambios que son parches o bug fixes.

Seguridad

La importancia de la seguridad dentro de nuestro proyectos, es parte de nosotros como desarrolladores. Debemos esforzarnos por asegurarnos de no introducir ningun software malisioso. Cuando nosotros descargamos un proyecto de npm, debemos de saber que npm incluye una herramienta para que podamos saber que paquete se esta instalando, y si alguno de ellos debiera ser actualizado. Por ejemplo, si instalamos un paquete que puede llegar a tener vulnerabilidades de seguridad, nos los va a marcar. En ese caso podemos ejecutar el comando npm audit. Este comando nos va a permitir auditar o examinar nuestro proyecto, y ver si hay algun cambio que pueda afectar nuestro proyecto.

Tambien podemos obteneer la informacion de nuestro proyecto auditado, en formato JSON, con el comando npm audit --json. Este comando nos genera un documento JSON, donde esta la informacion un poco mas detallada, de los que esta ocurriendo con nuestros paquetes.

Hay un comando disponible, que es el comando npm audit fix. Este comando nos permite solucionar los errores de vulnerabilidad y seguridad.
